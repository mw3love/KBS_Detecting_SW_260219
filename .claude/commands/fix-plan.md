Fix 수정계획 문서를 생성합니다.

인수 형식: `문제 설명` (예: `/fix-plan ROI 저장 후 사라지는 버그`)

**아래 단계를 순서대로 실행하세요:**

## 1단계: 인수 및 날짜 확인

`$ARGUMENTS`가 비어 있으면 어떤 문제인지 사용자에게 물어보고 중단하세요.

오늘 날짜를 YYMMDD 형식으로 확인하세요:
```bash
date +%y%m%d
```

파일명 핵심단어는 `$ARGUMENTS`를 2~5글자로 압축합니다 (예: ROI저장버그, 감지중단, 알림오작동).

## 2단계: 관련 코드 탐색

`$ARGUMENTS`를 바탕으로 관련 파일을 읽고 원인을 분석하세요.

- Grep으로 관련 키워드 검색
- 의심되는 파일 Read로 확인
- 원인 후보를 1순위, 2순위 순으로 정리

## 3단계: 수정계획 문서 작성

`Fix/YYMMDD_핵심단어_수정계획.md` 파일을 아래 구조로 작성하세요.

같은 날짜 파일이 이미 있으면 다른 핵심단어를 사용하세요.

---

문서 구조:

```
# [문제 제목] 수정계획

## 문제 현상
- 어떤 상황에서 발생하는지
- 재현 조건 (항상 / 간헐적 / 특정 조건)
- 사용자가 목격한 증상

## 원인 분석

### 1순위: [원인명]
- 파일: `경로/파일.py` L번호
- 설명: ...

### 2순위: [원인명] (있을 경우)
- ...

## 수정 대상 파일

| 파일 | 수정 위치 | 내용 |
|------|-----------|------|
| `파일.py` | `함수명()` | 설명 |

## 단계별 수정 계획

> 각 Phase는 새로운 대화세션에서 시작한다.
> 중단되어도 다음 세션에서 이 파일을 열어 이어서 진행 가능.

### Phase 1 — [작업 제목]
- [ ] 작업 항목
- [ ] 작업 항목
- [ ] 커밋: `메시지`

### Phase 2 — [작업 제목] (해당 시)
- [ ] 작업 항목
- [ ] 커밋: `메시지`

### Phase N — 검증
- [ ] 현장 테스트
- [ ] 로그 확인

## 참고
- 관련 설계 원칙: (core/CLAUDE.md 또는 ui/CLAUDE.md 중 해당 섹션)
- 관련 이력: (과거 유사 버그 있으면 기록)
```
