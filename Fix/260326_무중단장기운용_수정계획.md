## 기존 문서에서 누락된 항목 (신규 발견)

### 🔴 누락 1 — [HIGH] Windows GDI 핸들 누적 가능성

**검토 결과**: 기존 LONGEVITY 문서에서 Qt paintEvent의 GDI 핸들 관리를 검토하지 않음.

- **파일**: `kbs_monitor/ui/video_widget.py`, `kbs_monitor/ui/top_bar.py`
- **메커니즘**: Qt가 매 프레임마다 `QImage → QPixmap` 변환 시 내부적으로 Windows GDI 비트맵 핸들 생성/해제.
  - 30fps × `_render()` 호출 = 초당 30회 GDI 핸들 생성
  - `LevelMeterBar.set_level()` → `self.update()` = 오디오 스레드 ~43Hz × 2채널
- **위험**: Windows 프로세스의 GDI 핸들 상한은 기본 10,000개 (레지스트리 `GDIProcessHandleQuota`).
  미세 누수 시 2~3일 만에 한계 도달 → UI 렌더링 실패 → 프로그램 비정상 동작.
- **확인 방법**: 운용 중 작업 관리자 → 상세정보 탭 → GDI 개체 열 추가 → 수치 증가 여부 관찰
- **현재 판정**: Qt는 일반적으로 GDI 핸들을 올바르게 해제하나, QLabel.setPixmap()의 내부 캐시에서
  누수 사례가 보고됨 (Qt 버그 트래커). **2~3일 문제의 원인 후보 1순위**.

> 기존 LONGEVITY 문서에서 Qt 렌더링 GDI 핸들 검토를 완전히 누락함.

---

### 🟡 누락 2 — [MEDIUM] Python 프로세스 메모리 단편화

**검토 결과**: 기존 문서 미검토 영역.

- **메커니즘**: 200ms마다 numpy 배열 생성/해제 반복 (detector.py ROI resize, float32 변환).
  Python의 pymalloc은 메모리를 OS에 즉시 반환하지 않고 내부 free-list에 보관.
  장기 운용 시 프로세스의 실제 메모리 사용량이 점진적으로 증가.
- **증상**: 기능 장애 없음, 성능 저하. 저사양 PC(RAM 4GB 미만)에서는 OOM 유발 가능.
- **현재 판정**: 중간 위험도. 정기 재시작으로 자연 해소됨.

---

### 🟡 누락 3 — [LOW] PortAudio 글로벌 상태 누적

**검토 결과**: LONGEVITY 8.7에서 재연결 로직 자체는 검토했으나 PortAudio 글로벌 상태는 미검토.

- **메커니즘**: `sounddevice`의 PortAudio 백엔드는 프로세스 레벨 글로벌 상태 유지.
  오디오 장치 재연결 시 `sd.RawInputStream` 재생성 → PortAudio 컨텍스트 부분 재초기화.
  드라이버 구현에 따라 내부 리소스가 완전히 정리되지 않을 수 있음.
- **현재 판정**: 낮은 위험도. 재연결 빈도가 낮으면 실질적 영향 없음.
  정기 재시작으로 PortAudio 컨텍스트 완전 초기화 보장됨.

---

### 🟡 누락 4 — [LOW] video_capture.py 8시간 주기 재연결 미기재

**검토 결과**: 기존 LONGEVITY 문서 작성 후 추가된 기능으로 문서에 미기재.

- **현재 코드**: `_PERIODIC_RECONNECT_INTERVAL = 8 * 3600` (video_capture.py L10)
- **기능**: 8시간마다 DirectShow 캡처 장치를 강제 재연결하여 OpenCV COM 객체 누적 예방.
- **부작용**: 재연결 중 10~100ms 프레임 손실 (감지 1~3 주기 누락). 녹화 미포함.
- **현재 판정**: 양호한 설계. 로그 모니터링 시 "포트 N 정기 재연결 (freeze 예방)" 메시지 참고.

> **기존 LONGEVITY 최종 요약 및 모니터링 권장 사항 섹션에 이 기능을 추가해야 함**.

## 원인 분석

### 1순위: Windows GDI 핸들 누적 (또는 Qt 내부 핸들 누수)

- **파일**: `kbs_monitor/ui/video_widget.py` L93~111 (`_render()`), `kbs_monitor/ui/top_bar.py` (`LevelMeterBar`)
- **2~3일 타임라인 추정**: 30fps × 86400s × 3일 ≈ 780만회 GDI 핸들 생성/해제. 10,000개/회당 1개 누수 시 3일 후 한계 도달.
- **실제로는 Qt가 대부분 올바르게 해제하나, Windows + PySide6 특정 버전 조합에서 누수 사례 존재**

### 2순위: 프로세스 레벨 메모리 단편화 + OS 리소스 누적

- OpenCV DirectShow: 8시간 재연결이 있으나 COM 객체 잔류 가능성 (OS 레벨)
- Python GC + numpy 단편화: 기능 장애 아닌 성능 저하
- PortAudio 글로벌 상태: 재연결 반복 시 점진적 누적

### 결론

코드 레벨의 수정만으로는 **OS 레벨 리소스**(GDI 핸들, COM 객체, PortAudio 컨텍스트)의 장기 누적을 완전히 방지하기 어렵다. **가장 확실한 해결책은 정기적 프로세스 재시작**이다.

## 재시작 방식 비교

| 방식 | 장점 | 단점 | 권장도 |
|------|------|------|--------|
| A. 외부 Task Scheduler | OS 보장, 단순 | 각 PC별 수동 설정, 의존성 | 보조 수단 |
| **B. 내부 스케줄 재시작** | 코드에 내장, 자동 | 구현 필요, 10~30초 다운타임 | **1순위** |
| C. 소프트 리스타트 (상태 초기화) | 다운타임 없음 | GDI/COM/PortAudio 미정리, 불완전 | 비권장 |

**권장: B. 내부 스케줄 재시작 (subprocess + quit)**

- `QTimer` 1분 주기로 시각 확인 → 새벽 3시 0분에 `subprocess.Popen`으로 새 프로세스 시작 → 현재 프로세스 `QApplication.quit()`
- 프로세스가 완전히 종료/재시작되므로 GDI, COM, PortAudio, Python 메모리 모두 초기화
- 재시작 소요 시간: ~10~30초 (설정 로드 + 캡처 장치 연결 포함)
