## 2. 식별된 버그 4가지

### Bug 1: PREP-DBG 루프 조기 종료

**로그 증거 (27일)**:
```
00:30:01  [PREP-DBG] V2: ... still_timer=0.0s[기준60s] changed=0.0%[블록기준10.0%] 지속리셋: 0.0s전 [준비중/resolve=2/start=True]
```
- 이후 PREP-DBG 로그 **완전히 없음**
- 26일에는 같은 시간대에 수십 건의 PREP-DBG가 찍혔음
- 1건 기록 직후 루프가 탈출하거나 준비모드 판정 조건을 벗어난 것으로 추정

### Bug 2: still_timer 미누적 (61분간 changed=0.0%인데 timer=0.0s)

**로그 증거 (27일)**:
```
00:30:01  still_timer=0.0s  changed=0.0%
```
- `changed=0.0%`는 V2 프레임이 완전히 정지 상태임을 의미
- 정상이라면 매 200ms마다 still_timer가 누적되어야 함
- 61분간 timer가 0.0s에서 움직이지 않음 → **누적 로직이 실행되지 않음**

### Bug 3: 텔레그램 알림 미발송

- 정파모드 진입(02:00) 및 해제 시 텔레그램 발송 없음
- 원인: 경보중 상태를 거치지 않고 진입한 경우 알림 조건 미충족
- 또는 PREP-DBG 루프 종료로 인해 signoff_manager가 정상 흐름을 타지 못한 것

### Bug 4: 정파모드 진입 후 V2/V4 알림 억제 미적용

- 26일: V2 정파모드 진입 후 V2/V4/V6/V8 채널 알림 자동 억제
- 27일: 정파모드 진입 후에도 V2/V4 알림 억제 없음 → 정파 화면에 대한 블랙/스틸 오경보 가능

## 3. 근본 원인 추정

**핵심 단서**: 27일 00:30:01 로그에 `resolve=2` 표시

```
[준비중/resolve=2/start=True]
```

26일에 정파 사이클이 2회 완료된 후(`resolve=2`), 내부 상태가 완전히 초기화되지 않은 채로
27일 준비모드에 재진입한 것으로 추정된다.

**추정 흐름**:
1. 26일 정파 사이클 2회 완료 → `resolve=2` 상태 유지
2. 27일 00:30 준비모드 진입 시 → `resolve=2` 조건에서 "이미 처리 완료" 판정
3. still_timer 누적 조건 진입 차단 → timer=0.0s 고정
4. PREP-DBG 루프 1건 후 탈출

**결론**: `signoff_manager.py`의 준비모드(PREPARATION) 재진입 시 `resolve` 카운터 또는
관련 상태 플래그가 초기화되지 않아 감지 루프가 즉시 종료됨.

## 4. 수정 방향

1. **signoff_manager.py**: 정파모드 해제(`NORMAL` 복귀) 시 `resolve` 카운터 및 관련 상태 변수 완전 초기화
2. **still_timer 누적 조건 검토**: `resolve > 0`일 때 누적 로직이 건너뛰어지는 코드 경로 확인
3. **PREP-DBG 루프 탈출 조건**: 준비모드에서 루프 탈출하는 조건문 재검토 (특히 `start=True` + `resolve=N` 조합)
4. **텔레그램 알림 조건**: 경보중 경유 없이 정파모드 진입 시에도 알림 발송되도록 보완

## 6. 용어 해설

### `resolve`

**정파 사이클 완료 횟수를 세는 카운터** (`signoff_manager.py` 내부 변수)

- 정파모드(`SIGNOFF`) 상태가 한 번 완전히 끝날 때마다 +1 된다.
- "정파 시작 → 경보중 → 정파모드 → 방송 재개(NORMAL 복귀)" 사이클이 1회 완료되면 `resolve=1`, 2회면 `resolve=2`.
- **버그 관련성**: 26일에 정파 사이클 2회가 끝나 `resolve=2`가 된 상태에서, 프로그램이 완전 재시작되지 않고 27일 새벽에 다시 준비모드에 진입했다. 코드가 `resolve=2`를 보고 "이미 처리 완료된 사이클"로 잘못 판단하여 `still_timer` 누적을 건너뛴 것으로 추정된다.

### `[PREP-DBG]`

**준비모드(PREPARATION) 감지 루프가 실행 중임을 확인하는 디버그 로그** (`detector.py` 출력)

- 정파 준비 조건을 검사하는 루프가 200ms마다 돌 때마다 출력된다.
- **정상(26일)**: 00:44~00:47 사이에 수십 건 연속 출력 → 루프가 계속 돌며 `still_timer` 누적.
- **비정상(27일)**: 00:30:01에 1건 출력 후 완전히 없어짐 → 루프가 첫 실행 직후 탈출했다는 증거.
- `[PREP-DBG]`가 찍히지 않으면 준비모드 감지 자체가 동작하지 않는 것이며, 이것이 `still_timer=0.0s` 고착의 직접적 원인이다.
