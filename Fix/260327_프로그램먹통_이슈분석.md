## 3. 로그에서 발견된 추가 단서

### 단서 1: 09:29 두 번째 정파준비모드 — PREP-DBG 없이 즉시 종료

```
09:29:22  [INFO] SIGNOFF - 2TV 정파준비모드를 시작합니다
09:30:06  [INFO] SIGNOFF - 2TV 정파준비모드를 종료합니다
```

- PREP-DBG 로그가 단 **0건** → 루프 진입 즉시 탈출
- 00:30의 버그(PREP-DBG 1건 후 탈출)보다 더 심각하게 퇴화
- `resolve=2` 상태에서 재진입 시 루프 조건 자체를 통과하지 못하는 것으로 추정

### 단서 2: 09:39:44 텔레그램 오류로 프로그램 종료

```
09:39:44  [ERROR] TELEGRAM - 전송 처리 중 예외 (스레드 유지): TypeError: 'NoneType' object is not subscriptable
```

- `TelegramNotifier` 내부에서 None 객체에 subscript 접근 시도
- 이 오류 직후 프로그램이 종료됨 → 해당 오류가 종료의 직접 원인이거나, 연쇄적으로 다른 곳에서 크래시 발생
- 먹통 상태에서도 텔레그램 큐에 메시지가 쌓였다가 오류를 일으킨 것으로 추정

### 단서 3: 09:33 채널그룹 재설정 후 resolve 초기화

```
09:33:27  DIAG-SIGNOFF - 그룹1=[1TV/IDLE/감시:V1/억제:V1,V2,V3,V4] 그룹2=[2TV/IDLE/감시:V5/억제:V5,V6,V7,V8]
```

- 재설정 후 `resolve=0/start=N`으로 일부 채널이 초기화됨
- 그러나 먹통은 해소되지 않음 → resolve 재설정만으로는 부족

## 4. 근본 원인 가설

### 가설 A: resolve 미초기화로 인한 signoff_manager 상태 오염 (기존 Bug 분석과 동일)

00:30에 PREP-DBG 루프가 조기 탈출된 이후, `signoff_manager`가 내부적으로 비정상 상태에 머물렀고, 이것이 시간이 지남에 따라 다른 컴포넌트(알람, 감지 타이머 등)에도 영향을 미쳤을 가능성

### 가설 B: 장시간 운영 중 스레드/큐 상태 누적 오염

04:30에 정파모드가 해제된 후, 내부 상태가 완전히 초기화되지 않은 채 4~5시간이 경과하면서 타이머/큐/스레드 상태가 누적 오염되었을 가능성

### 가설 C: 09:39:44 오류가 실제 먹통의 원인이 아닌 결과

텔레그램 오류는 **먹통 상태의 결과**로 발생했을 수 있음. 즉, 먹통이 먼저 발생 → 비정상 큐 상태 → 텔레그램 오류 순서일 가능성
