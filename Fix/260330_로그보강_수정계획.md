# 장기운용 먹통 재발 시 원인 추적 — 로그 보강 수정계획

## 문제 현상

- **상황**: eval-freeze로 먹통 원인 수정 후, 실운용 중 동일 문제가 재발할 경우 log.txt만으로 원인을 파악해야 함
- **재현 조건**: 장기 운용(수 시간~수일) 후 간헐적 발생
- **현재 로그의 한계**:
  - 감지 루프 heartbeat가 **5분(1500카운트)마다 1회**뿐 → 먹통 발생 후 최대 5분 뒤에야 log에서 확인 가능
  - QTimer 활성 여부를 로그로 확인 불가 (타이머 자체가 멈췄는지 판단 불가)
  - VideoCaptureThread 프레임 수신 정상 여부 추적 불가
  - 로그 포맷에 스레드 이름이 없어 멀티스레드 문제 구분 어려움

## 원인 분석

### 1순위: 감지 루프 heartbeat 간격 과다 (5분)
- 파일: `kbs_monitor/ui/main_window.py` L265
- `self._detection_count % 1500 == 0` → 200ms × 1500 = 5분
- 먹통 발생 시 로그에서 "마지막 heartbeat 이후 5분이 지났는지" 확인이 늦음
- **개선**: 1500 → 150 (30초마다)

### 2순위: QTimer 활성 상태 로그 없음
- 파일: `kbs_monitor/ui/main_window.py` L265~275
- 현재 heartbeat에 각 타이머의 `isActive()` 상태가 포함되지 않음
- 타이머가 예외 등으로 중단됐을 때 실시간 확인 불가
- **개선**: heartbeat 로그에 `detect/summary/restart` 타이머 상태 포함

### 3순위: VideoCaptureThread 프레임 heartbeat 없음
- 파일: `kbs_monitor/core/video_capture.py` L51~
- `frame_count` 변수 없음, 오류 발생 시에만 로그 기록
- 프레임이 정상 수신되는지 추적 불가
- **개선**: frame_count 카운터 추가, 500프레임(약 16초)마다 DEBUG 로그

### 4순위: 로그 포맷에 스레드 이름 없음
- 파일: `kbs_monitor/utils/logger.py` L50~53
- 현재: `"%(asctime)s [%(levelname)s] %(message)s"`
- **개선**: `"%(asctime)s [%(levelname)s][%(threadName)s] %(message)s"`

> **확인됨 — 수정 불필요**: `signoff_manager.py`의 PREP-DBG 로그는 이미
> `still_trigger_sec` 기준값을 포함하고 있음 (L593, L600, L611)
