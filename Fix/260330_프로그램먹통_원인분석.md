## 먹통 원인 가설

### 가설 1: SIGNOFF→IDLE 복귀 시 enter 타이머 미초기화로 인한 다음 사이클 PREPARATION 즉시 탈출

- **관련 위험 요소**: C1, H1
- **발생 시나리오**:
  1. 1일차 자정: 정파 사이클 정상 완료 (PREPARATION → SIGNOFF → IDLE)
  2. `_transition_to(gid, IDLE)` 호출 시 `_reset_enter_timers()`가 **호출되지 않음**
  3. `_video_enter_start[gid]`에 1일차 스틸 감지 시작 시각(예: 00:30:01의 Unix timestamp)이 그대로 잔류
  4. 2일차 자정 동일 시각: PREPARATION 재진입
  5. `_tick_preparation()` 최초 호출 시 `v_elapsed = now - _video_enter_start[gid]` = 약 86400초(하루) >> `still_trigger_sec`(60초)
  6. **즉시 SIGNOFF 전환** → PREPARATION 구간 0초 유지 → PREP-DBG 로그 0건
  7. 만약 스틸이 실제로 감지되지 않는 시각이라면 `_video_enter_start`가 `None`일 수 있으나, 1일차 정파 감지 중 스틸이 지속되다 타이머 만료(SIGNOFF 전환)로 종료된 경우 `_reset_enter_timers()`가 호출되어 `None`으로 리셋됨. 문제는 이와 **별개로** 1일차 스틸 감지 중에 IDLE로 직접 복귀하는 경로(`end_time` 도달 또는 `exit_released` 경로)에서 `_reset_enter_timers()`가 누락된 것임.
- **근거**: `_transition_to()` 라인 754~756에서 `IDLE` 진입 시 `_manual_override`와 `_preparation_entered_at`만 초기화하고, `_video_enter_start` / `_video_enter_not_still`은 초기화하지 않는다. `_reset_enter_timers()`는 `_tick_impl()` 내 PREPARATION→IDLE 경로(라인 538)와 `cycle_state()` 내(라인 333)에서만 호출된다. SIGNOFF→IDLE 경로(라인 554)는 `_transition_to()` 직접 호출이며 enter 타이머 초기화 없음.
- **확신도**: 높음 — 이전 `260327_정파감지실패_버그분석.md`의 PREP-DBG 0건 재현과 정확히 일치하는 메커니즘. 단, 1일차에 `_video_enter_start`가 non-None 상태로 잔류하는 조건(스틸 감지 시작 후 SIGNOFF 진입 없이 end_time 도달하는 경우)이 필요.

### 가설 2: _play_sound Event 상태 오염으로 인한 알림음 영구 무음

- **관련 위험 요소**: H2
- **발생 시나리오**:
  1. 정상 알림음 재생 중 `_sound_thread`가 내부 예외(WAV 파일 오류, PortAudio 예외 등)로 종료
  2. 예외 종료 시점에 `_stop_sound`가 `clear()` 상태이거나 루프 내 `wait()` 중 예외 발생으로 `_stop_sound`는 `set()` 되지 않음
  3. 새 알림 발생 시 `_play_sound()` 내 `is_alive()` == False 확인 → `join(0.5)` 후 `_stop_sound.clear()` → 새 `_play_sound_worker` 스레드 생성
  4. 그러나 **`_stop_sound`가 이미 `set()` 상태**라면 `clear()` 호출 전에 발생한 이전 스레드의 `set()` 이후, `clear()` 이전에 새 스레드가 `is_set()` 체크를 통과하면 즉시 루프 탈출
  5. 특히 `_play_sound()` 라인 250: `self._stop_sound.clear()` → 라인 251: 새 스레드 생성 → 스레드 내 첫 번째 `while not self._stop_sound.is_set()` 체크. 이 사이에 `_stop_playback()`이 외부에서 호출되면 `set()`되어 스레드 즉시 탈출.
  6. 결과: 알림 발생 시 소리 없음 (시각 깜박임은 정상, 알림 기능 자체는 있는 것처럼 보이나 소리 없음)
- **근거**: `alarm.py` 라인 240 `if self._sound_thread and self._sound_thread.is_alive(): return` — 죽은 스레드 재생성은 있음. 그러나 `_stop_sound`는 `threading.Event()` 인스턴스를 재사용하며 `play_test_sound()`와 달리 `_play_sound()`는 새 `Event` 객체를 생성하지 않고 `clear()`만 수행(라인 250). 따라서 `set()`이 잔류한 상태에서 `clear()`-`스레드 시작` 사이에 경쟁 조건 발생 가능.
- **확신도**: 중간 — 경쟁 조건이 발생하려면 타이밍이 맞아야 하므로 재현이 어렵지만, 장기 운영 시 발생 가능성 존재. 운영자가 알림음 테스트 버튼도 무반응이라고 보고했는데, 테스트는 `play_test_sound()`를 사용하므로 새 `Event` 객체를 생성 → 이 경로는 안전. 따라서 실제 먹통에서 테스트 버튼까지 무반응인 경우는 이 가설보다 가설 1(또는 다른 원인)이 주요 원인일 가능성이 높음.

### 가설 3: _run_detection() try-except 외부 초기화 코드에서의 예외로 인한 감지 루프 일시 중단

- **관련 위험 요소**: H3
- **발생 시나리오**:
  1. `_run_detection()` 내 try 블록 이전(라인 257~374, DIAG 로그 출력 구간)에서 `self._detector._last_raw.items()` 등의 dict iteration 중 크기 변경(다른 스레드의 `update_roi_list()` 동시 호출)으로 `RuntimeError: dictionary changed size during iteration` 발생
  2. 해당 예외는 **try 블록 외부**(라인 257~374)에서 발생하므로 catch되지 않음
  3. `_run_detection()` 전체가 예외로 종료 → 타이머는 유지되나 해당 호출은 실패
  4. `_detection_count % 1500 == 0` 조건(5분마다)에서만 DIAG 로그가 실행되므로, 5분 주기로 예외가 발생할 가능성이 있음. 단, 5분 주기 한 번만 실패하고 다음 주기는 정상이라면 먹통이 아님.
- **근거**: `main_window.py` 라인 277 `for lbl, raw in self._detector._last_raw.items():` — 이 dict는 `detect_frame()`에서 ROI별로 수정된다. `_run_detection()` 자체가 단일 스레드(Qt 이벤트 루프)에서 실행되므로 동시 수정은 발생하지 않는다. 따라서 이 가설의 실제 위험도는 낮음. 예외 발생 가능성은 거의 없다.
- **확신도**: 낮음 — Qt 단일 스레드 모델상 dict 동시 수정은 불가능. 단, try 블록 외부 코드가 존재한다는 구조적 문제는 향후 리팩터링 시 주의 필요.

   - 기대 효과: 다음 정파 사이클 PREPARATION 진입 시 `_video_enter_start`가 항상 `None`으로 시작 → PREP-DBG 0건 즉시 탈출 버그 해소

2. **C2 — `_transition_to()` IDLE 복귀 시 `_dbg_prev_still` 초기화 누락**
   - 대상: `core/signoff_manager.py` `_transition_to()` 라인 754~756
   - 수정 방향: `new_state == SignoffState.IDLE` 분기에 `self._dbg_prev_still[group_id] = None` 추가
   - 기대 효과: 다음 사이클 PREPARATION 진입 시 스틸 감지 시작 로그가 정확히 출력됨

### 추가 검토 필요 (High)

1. **H2 — `_play_sound()` 내 `_stop_sound` Event 재사용 경쟁 조건**
   - 확인 사항: `_play_sound()`에서 `_stop_sound.clear()` 대신 `threading.Event()` 새 인스턴스로 교체하는 방안 검토
   - 주의: `play_test_sound()`는 이미 새 Event 객체를 사용하므로 안전. `_play_sound()`만 수정 대상.
   - 단, 이 버그는 실제 먹통 재현 조건이 좁으므로 로그 확인 후 판단.

2. **H3 — `_update_summary()` 및 `_check_scheduled_restart()` try-except 보호 부재**
   - 확인 사항: 두 함수에 `try-except` 추가 또는 각 함수 내부에서 예외 발생 시 파일 로그만 기록하고 계속 진행하도록 보호
   - 우선순위: `_check_scheduled_restart()`는 파일 IO를 매 10초 수행하므로 예외 가능성 있음. `_update_summary()`는 단순 UI 업데이트이므로 낮음.

### 참고 (Low)

1. **L1** — `force_still_labels` 사용 시 near-miss 추적 누락은 DIAG 전용 기능이므로 운영에 영향 없음. 추후 리팩터링 시 정리.
2. **L2** — `_acknowledged_alarms` 집합은 현재 설계에서 무한 성장 없음. 유지.
3. **L3** — `_last_sent` 정리 조건 (`len > 50`)을 시간 기반으로 교체하거나, 채널 수 × 감지 타입 수를 기준으로 조정 가능.
4. **M2** — `_record_worker` 무한 루프 가능성: 캡처 스레드 연결 끊김 감지 후 `push_frame()`이 호출되지 않는 상황을 대비해 `_record_worker` 내에 타임아웃(예: `_record_end + 30초` 초과 시 강제 종료) 추가 검토.
