|---|------|------|----------------|------|
| H1 | E | `telegram_notifier.py` | `_send()` L352~356 | **비-429 HTTP 오류 시 `_consecutive_failures` 미증가**. `resp.status_code`가 403, 500 등일 때 `return False`만 하고 카운터를 증가시키지 않음. 이 경우 서버 측 오류가 반복되면 `_log_with_suppression` 경로를 타지 않아 로그 스팸이 억제되지 않음. 단, 현재 이 경로의 로그는 `self._log(..., error=True)`를 직접 호출하므로 매번 UI에 표시됨. 계획서에서 "오류 로그 스팸 억제"를 목표로 했으나 이 경로는 억제 대상에서 빠짐. |

### Medium

| # | 영역 | 파일 | 위치(함수/라인) | 설명 |
|---|------|------|----------------|------|
| M1 | E | `telegram_notifier.py` | `_send()` L357~372 | **재시도 중간 로그가 억제 대상에서 제외됨**. `attempt < _SEND_RETRY_COUNT`일 때(L359~365) `self._log(f"전송 오류 (재시도 ...)", error=True)`를 직접 호출. 연속 실패 중(예: 50회 연속)에도 재시도 중간 로그 2건은 항상 UI에 표시됨. 최종 실패 로그만 억제되고 중간 로그는 통과. 방화벽 환경에서 알림 10건 발생 시 재시도 중간 로그 20건은 여전히 UI에 노출됨. |
| M2 | E | `telegram_notifier.py` | `_log_with_suppression()` L273 | **`_consecutive_failures`가 0일 때 `n <= 3` 조건에 해당하여 UI 로그 출력**. `_log_with_suppression`은 실패 카운터 증가 후 호출되므로 `n=0`인 경우는 정상 흐름에서 발생하지 않으나, 만약 성공 직후(`_consecutive_failures = 0`) 외부 catch 경로에서 `_log_with_suppression`이 호출될 경우 `n=0`이 됨. 실질적 영향은 없으나 방어적으로 `n <= 0` 케이스 명시 고려. |
| M3 | D | `telegram_notifier.py` | `_send()` L368, L374 | **`_consecutive_failures` 이중 증가 가능성**. `_send()` 내부에서 카운터를 증가시키고 `return False`를 반환하는데, `_worker_loop()`의 `success = self._send(item)` 후에는 `if success: reset`만 있고 실패 시 카운터 증가가 없으므로 이중 증가는 발생하지 않음. 다만 `_send()`가 예외를 던지면(현재 코드에서는 모두 catch하므로 불가) `_worker_loop` 외부 catch에서 추가 +1이 됨. 현재 코드에서는 문제 없으나, 향후 `_send()` 수정 시 주의 필요. |

## 권장 조치

### 추가 수정 필요 (High)

1. **H1 해소**: `_send()` L352~356의 비-429 HTTP 오류 경로에서도 `_consecutive_failures += 1` 및 `_log_with_suppression()` 적용. 현재:
   ```python
   self._log(f"전송 실패 {resp.status_code}: {resp.text[:120]}", error=True)
   return False
   ```
   권장:
   ```python
   self._consecutive_failures += 1
   self._log_with_suppression(f"전송 실패 {resp.status_code}: {resp.text[:120]}")
   return False
   ```

### 검증 방법

1. 잘못된 `bot_token`으로 텔레그램 활성화 → 알림 10건 이상 발생시키기 → UI 로그에 1~3회만 오류 표시되고 이후 10회 배수에서만 요약 표시되는지 확인
2. `configure()` 재호출 후 카운터가 리셋되는지 확인 (설정 저장 후 다시 실패 → 1~3회 UI 표시 재개)
3. 429 Rate Limit 시뮬레이션은 현장에서 어려우므로 코드 리뷰로 대체
