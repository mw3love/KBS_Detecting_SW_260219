"10의 배수회"가 아이템 기준인지 예외 발생 횟수 기준인지 불명확하여, 실제 "10번째 알림마다 요약"이 의도와 다르게 동작할 수 있음. |

## 권장 수정사항

### 계획서 수정 필요 (Critical/High)

1. **[C1] Phase 1에 `status_changed` 연결 추가 명시**
   - `_start_threads()`에서 `self._capture_thread.status_changed.connect(lambda msg: self._logger.info(f"SYSTEM - {msg}"))` 연결을 Phase 1 작업 목록에 추가해야 함
   - 현재 `video_capture.py`의 상세 메시지(연결 성공/실패/신호없음)가 UI에 전혀 표시되지 않고 있어, `disconnected` lambda를 슬롯으로 교체해도 `status_changed` 연결 없이는 "포트 N 신호 없음" 메시지가 노출되지 않음

2. **[H1] Phase 1에 `connected` lambda 교체도 포함**
   - `disconnected` lambda와 함께 `connected` lambda(L207)도 `_active_capture_port` 기반 슬롯으로 교체하여 클로저 문제를 일관되게 해소해야 함

3. **[H2] Phase 2에 스레드 안전성 고려 및 카운터 증가 기준 명확화**
   - `_consecutive_failures`는 워커 스레드에서만 접근(읽기/쓰기)하도록 제한하거나, `threading.Lock()`으로 보호
   - 429(Rate Limit) 케이스에서의 카운터 처리 방침 명시: 재시도 포함 최종 실패 시 1회만 증가 vs. 예외 발생마다 증가

### 계획서 보완 권장 (Medium/Low)

1. **[M1] "파일 전용 로그" 구현 방법 구체화**
   - `_log(msg, error=False)` 호출 시 `_log_info`(`file_only`)를 사용하는 것이 UI 억제가 됨을 계획서에 명시, 또는 별도 `_log_file_only()` 내부 메서드 추가 방법 제시 필요

2. **[M2] `_consecutive_failures` 증가 단위 명확화**
   - "N회 연속 실패"를 발송 아이템 단위로 카운트할지, 예외 발생 횟수 단위로 카운트할지 명시
   - 아이템 단위가 운영자 체감에 더 부합하므로 `_send()` 완료 시점(루프 전체 실패 = 1회 증가) 방식 권장

3. **[L1] `_active_capture_port` 초기값 명시**
   - `__init__`에서 `self._active_capture_port: int = self._config.get("port", 0)` 초기화 구문을 계획서 Phase 1 체크리스트에 추가

## 종합 판정

- **구현 가능 여부**: 수정 후 가능
- **Critical 해소 필요**: `status_changed` 시그널 연결 누락(C1) — Phase 1 수정 내용의 효과가 이 연결 없이는 절반만 발현됨. 반드시 계획서에 추가 후 구현.
- **특별 주의사항**:
  - Phase 1의 핵심 가치는 "포트 번호 정확도"보다 "현재 미연결된 `status_changed` 시그널 노출"에 있음. 계획서 원인 분석(L19~21)에서 lambda 클로저 문제를 언급했지만 `status_changed` 연결 자체가 빠져 있어 실제 수정 범위가 더 크다.
  - Phase 2는 기능 자체는 비교적 단순하나, 카운터 증가 단위 불명확 문제(M2)를 해소하지 않으면 "10의 배수회" 로직이 의도와 다르게 동작할 수 있음. 구현 전 단위 결정 필요.
