# 프로그램 먹통 원인 분석

**분석일**: 2026-04-05
**에이전트**: eval-freeze (analyze 모드)
**대상 버전**: `caa3a3c v1.6.12: 버전 번호 및 About 날짜 업데이트`
**분석 요청 범위**: `_run_detection()` DIAG 블록 집중 + 텔레그램/알림음 테스트 버튼 먹통 가능 원인

---

## 배경 및 분석 범위

이 분석은 `analyze` 모드이지만, 이미 `260405_감지루프중단_수정계획.md`가 작성된 이후 요청된 것이므로
기존 분석과의 중복을 최소화하고 아래 5개 확인 포인트에 집중한다.

1. DIAG 블록(L284~L401) 중 예외 발생 가능 위치 — 특히 L360~390 DIAG-AUDIO 섹션
2. `_last_detection_time` 이동 후 `_latest_frame is None` 조기 반환 케이스에서 health check 오동작 가능성
3. 텔레그램/알림음 테스트 버튼 먹통의 다른 가능한 원인 (장기 실행 관련)
4. DIAG 블록 try-except 추가 시 traceback 로그 폭풍 방지 (30초마다 반복) — 로그 억제 필요 여부
5. `_detection_count` Python int 한계 가능성 확인

## 먹통 원인 가설

### 가설 1: DIAG-AUDIO 예외 → 감지 skip (부분 freeze)

- **관련 위험 요소**: C1, C2, H1, H2
- **발생 시나리오**:
  1. 프로그램 시작 후 정상 실행 중 (`_detection_count` 누적)
  2. `_detection_count % 150 == 0` 조건 충족 (30초마다)
  3. DIAG-AUDIO 섹션(L362~L390)에서 미상의 예외 발생
  4. 함수가 L283 직후 조기 종료 → L403 감지 try 블록 미실행
  5. 그러나 `_last_detection_time`은 L283에서 이미 갱신됨
  6. Health Check: `detect_stale` = False → 상단바 경고 없음
  7. 30초 후 다시 동일 예외 → 반복 (traceback 로그도 파일에 없음)
  8. wall-clock 기반 스틸/블랙 타이머는 149회/150회 실행으로 정상 동작하나 ROI 상태 갱신 지연
  9. 운영자 인지 불가 상태로 수십 시간 경과
- **근거**: `20260404.txt` 10:50:43에서 DIAG-SIGNOFF까지 출력 후 DIAG-AUDIO가 완전 소멸. 이후 13.5시간 동안 HB 로그(`SYSTEM-HB`)는 정상 출력 → QTimer + 메인 스레드 살아있음 확인. DIAG 블록 try-except 없는 구조(C2)가 직접적 증거.
- **확신도**: 높음 — 로그 패턴이 C2 구조 버그와 정확히 일치

### 가설 2: 텔레그램/알림음 테스트 버튼 먹통 (독립 버그)

- **관련 위험 요소**: H3, M3
- **발생 시나리오**:
  - **알림음 테스트**: `play_test_sound()` 호출 → `_play_test_worker` daemon 스레드 생성 → `winsound.PlaySound(SND_SYNC)` 호출. 이 경로에서 `threading.Event()` 새 객체가 생성되므로(CLAUDE.md 설계 원칙 준수) 이론적 문제 없음. 그러나 테스트 버튼이 무반응이면 버튼 클릭 이벤트 자체가 처리되지 않는 것으로 해석 가능.
  - **텔레그램 테스트**: daemon 스레드 + QTimer 폴링 방식 (v1.5.8 수정). 워커 스레드가 살아있고 `notify()`가 큐에 정상 삽입되는 한 문제없음.
  - **공통 가설**: 가설 1의 DIAG 예외가 텔레그램/알림음과 무관하다면, **테스트 버튼 먹통은 별도 원인**. 가능성: (a) 설정창(`SettingsDialog`)이 열려있고 버튼 클릭 이벤트가 설정창의 오버레이에 의해 차단. (b) 메인 스레드 Qt 이벤트 큐에 처리 대기 중인 항목이 많아 버튼 응답이 지연. (c) `winsound`가 시스템 오디오 장치 점유로 블록.
  - **미확인 질문**: freeze 상태에서 "테스트 중..." 레이블이 표시되는지 여부로 (a)/(b) vs (c) 구분 가능.
- **확신도**: 낮음 — 현재 코드에서 명확한 원인 식별 불가. Phase 1 traceback 후 조사 필요.

## 권장 조치

### 즉시 수정 (Critical — Phase 1, `260405_감지루프중단_수정계획.md` 연계)

1. **`_last_detection_time` 이동** (`ui/main_window.py` L283 → L403 try 블록 내부 첫 줄)
   - 수정 후: DIAG 예외 발생 시 5초 이내 Health Check 상단바 경고 표시
   
2. **DIAG 블록 독립 try-except 보호** (`ui/main_window.py` L284~L401 전체)
   - `import traceback`을 파일 상단(L1~L13)에 추가 (현재 L688에 지역 import만 존재)
   - except 블록에서 `_log.error()` + `traceback.format_exc()` 로그 후 정상 반환

### 추가 검토 필요 (High)

3. **DIAG traceback 로그 폭풍 억제** (H2)
   - 30초마다 동일 예외 반복 가능 → 24시간 2,880회 traceback 로그
   - 권장: DIAG 블록 첫 번째 예외 시에만 traceback 전체 출력, 이후 동일 예외는 억제
   - 구현 방법: `_diag_last_error_type: Optional[str] = None` 인스턴스 변수 추가
     ```python
     except Exception as _diag_e:
         err_type = type(_diag_e).__name__
         if err_type != self._diag_last_error_type:
             _log.error("DIAG 로깅 오류 (감지 계속): %s\n%s",
                        _diag_e, traceback.format_exc())
             self._diag_last_error_type = err_type
         else:
             _log.debug("DIAG 로깅 오류 반복 (동일 타입 억제): %s", _diag_e)
     ```
   - 또는 더 단순하게: `_diag_error_count` 카운터로 처음 1회 + 10회마다 1회 출력

4. **텔레그램/알림음 테스트 버튼 먹통 추가 진단** (M3, 가설 2)
   - Phase 1 배포 후 traceback으로 DIAG 예외 원인 확정
   - 다음 freeze 발생 시 "테스트 중..." 레이블 표시 여부 확인 요청 (운영자)
   - 결과에 따라 Phase 3 진행 (alarm.py `_play_test_worker` 또는 Qt 이벤트 처리 조사)

### 참고 (Low)

5. **`_detection_count` int 오버플로**: 걱정 불필요 (L1 — 확인 포인트 5 종결)
