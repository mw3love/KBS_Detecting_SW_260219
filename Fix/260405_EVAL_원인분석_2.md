## 기존 분석 문서와의 관계

이 문서는 `260405_EVAL_원인분석.md`의 **후속 분석**이다.
기존 문서에서 식별된 C1(DIAG 블록 `_last_detection_time` 위치), C2(DIAG 블록 try-except 미보호)는
**현재 코드에서 미해결 상태**이며, 이번 분석은 그 외 추가 위험 요소에 집중한다.

특히 이번 요청의 핵심 맥락:
- **정파 후 2~3일 뒤 발생** — 정파 사이클이 트리거가 될 가능성
- **텔레그램 고착** — "발송 중" 상태로 멈춤 (워커 스레드 hang)
- **테스트 버튼 무반응** — `threading.Thread.start()` 실패 또는 Qt 이벤트 블로킹
- **crash_report 없음** — 재시작 직전 내부 상태 미포착 → 진단 공백

  1. 프로그램 시작 후 정상 실행 중 (`_detection_count` 누적)
  2. `_detection_count % 150 == 0` 조건 충족 (30초마다)
  3. DIAG-AUDIO 섹션(L360~L390)에서 미상의 예외 발생 — traceback이 stderr에만 출력, 파일 로그에 없음
  4. 함수가 L284 직후 조기 종료 → L403 감지 try 블록 미실행 (30초마다 1회 skip)
  5. 그러나 `_last_detection_time`은 L283에서 이미 갱신됨 → Health Check `detect_stale` = False → 상단바 경고 없음
  6. 스틸/블랙 타이머는 wall-clock 기반이라 149/150회 실행으로 정상 동작하나, ROI 상태 dict 갱신이 30초마다 1회 지연
  7. 4월 4일 10:50:43 로그: DIAG-SIGNOFF 이후 DIAG-AUDIO 완전 소멸 → 이후 13.5시간 SYSTEM-HB는 정상 (Qt 이벤트 루프 생존 확인)

### 가설 2: OS 스레드 수 고갈 → 테스트 버튼 무반응 + 텔레그램 고착

- **관련 위험 요소**: C3, C4, H2, M4

  2. OS 스레드 수가 Windows 한계(기본 ~2000개, 프로세스당)에 근접
  3. 텔레그램 테스트 버튼 클릭 → `_on_telegram_test()` L941 `threading.Thread(...).start()` → `OSError: [WinError 1816]`
  4. PySide6 슬롯에서 unhandled exception → `_tg_test_timer.start(500)` L948 미실행 → "테스트 중..." 영구 고착
  5. 알림음 테스트 버튼 클릭 → `play_test_sound()` L172 `.start()` → 동일 `OSError` → 소리 없음
  6. 새 정파 시 텔레그램 알림 발송도 `notify()` → `is_alive()==False` 감지 → 재시작 시도 → 동일 `OSError` → 발송 불가
- **근거**: (a) `_on_telegram_test()` L941~L948에 try-except 없음(C3). (b) `play_test_sound()` L172 `.start()`에 try-except 없음(C4). (c) `260405_감지루프중단_수정계획.md`에서 "OS 스레드 자원 고갈 의심"으로 이미 기록. (d) Phase 1 SYSTEM-HB에 `threading.active_count()`가 추가되어 있지 않아 현재 스레드 수를 확인할 수 없음.
- **확신도**: 중간 (5/10) — 스레드 누수 원인이 구체적으로 확정되지 않음. Phase 1의 `threading.active_count()` 로그로 다음 freeze 시 확인 필요.

### 가설 3: 정파 사이클 후 `_latest_video` dict stale 값 → PREPARATION 타이머 오동작

- **관련 위험 요소**: M3
- **발생 시나리오**:
  1. 정파(SIGNOFF) 중 V5(enter_roi)의 `is_still=True`가 `_latest_video["V5"] = True`로 캐시됨
  2. SIGNOFF → IDLE 전환 후 ROI 목록이 변경(V5 삭제 또는 label 변경)
  3. `_latest_video`에는 여전히 `"V5": True` 잔류 (정리 로직 없음)
  4. 다음 날 PREPARATION 재진입 시, 새 enter_roi `v_label`이 변경 전 label이면 `_latest_video.get(v_label, False) = True`
  5. `_video_enter_start`가 즉시 갱신 → 빠른 SIGNOFF 조기 전환 가능
- **근거**: `_latest_video.update(still_results)`가 키를 누적하며 삭제 없음(M3). 그러나 일반적으로 ROI label은 V1~V16 고정이므로 실제 label 변경은 드문 경우.
- **확신도**: 낮음 (2/10) — ROI label 변경이 없으면 발생하지 않음. 운영 환경에서 label 변경 여부 불명.
