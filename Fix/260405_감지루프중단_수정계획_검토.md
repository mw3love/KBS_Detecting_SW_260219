## 권장 수정사항

### 계획서 수정 필요 (Critical/High)

1. **[C2] `_last_detection_time` 이동 위치 재검토**: Phase 1에서 `try` 블록 **내부**로 이동하면 감지 로직 자체 예외 시 false positive health alert가 발생한다. 올바른 위치는 DIAG 블록(`if % 150 == 0:`) **이후**, `try:` **직전** (즉 현재 L402와 L403 사이)이다.

   ```python
   if self._detection_count % 150 == 0:
       try:
           # 기존 DIAG 블록 전체
           ...
       except Exception as _diag_e:
           _log.error("DIAG 로깅 오류 (감지 계속): %s\n%s",
                      _diag_e, traceback.format_exc())

   self._last_detection_time = time.time()   # ← DIAG 이후, try 이전

   try:
       video_rois = ...
   ```

   이렇게 하면 DIAG 예외 발생 여부와 무관하게 `_last_detection_time`이 항상 갱신되며, 감지 로직 예외와 무관하게도 동일하다.

   단, 이 경우 계획서의 "DIAG 예외 → health check 탐지" 설계는 성립하지 않게 된다. Health Check 이동은 **기존의 다른 실패 경로**(예: 프레임 없음 즉시 return, roi_overlay return)와의 일관성을 위해 의미가 있으며, DIAG 예외 탐지와는 별개로 설명해야 한다.

2. **[H1] Health Check 이동의 효과 재기술**: Phase 1의 "DIAG 예외 발생 시 5초 후 health check 경고 표시"는 **사실과 다르다**. Health Check는 DIAG 예외(30초마다 1콜, 나머지 149콜 정상 갱신)를 탐지할 수 없다. Health Check 수정의 실제 효과는 "QTimer 완전 사망 또는 감지 루프 전체 hang 상황에서 운영자에게 경보"이며, 이것은 별도의 의미있는 안전장치이다. 계획서에서 두 가지를 분리하여 정확히 서술해야 한다.

3. **[C1] 로그 분석 결과와 원인 분석의 연결 보완**: "30초마다 1회 감지 skip → 감지에 영향 최소"는 사실이나, "13.5시간 DIAG-AUDIO 소멸"이 운영에 미치는 위험을 더 명확히 기술해야 한다. DIAG 블록 예외의 핵심 피해는 감지 skip이 아니라 **진단 로그 소멸**이다. 이것은 Phase 2 traceback 확보 후 근본 원인을 수정해도 남는 설계 취약점이므로, Phase 1의 try-except 보호가 핵심 수정임을 명기하면 좋다.

### 계획서 보완 권장 (Medium/Low)

1. **[M2] 예외 원인 후보 분석 재검토**: DIAG 블록은 try 블록(감지 로직) 이전에 실행되므로, `dict changed size during iteration`은 동일 콜 내에서는 발생할 수 없다. 실제 예외 원인은 `_audio_level_states.items()` 이터레이션 중 **dict의 크기 변경**이 아니라, 내부 값(DetectionState) 접근 시의 `AttributeError` 또는 `buf`가 빈 deque인 경우의 `ZeroDivisionError`(단, L365에 `if buf` 조건 존재 — 불가), 또는 format string 타입 오류일 가능성이 더 높다. Phase 2 진입 전 traceback 확인이 가장 중요하며, 예상 원인 목록에 이 점을 반영하면 좋다.

2. **[L1] DIAG 예외 로거 선택 명시**: except 블록에서 `_log.error()` 대신 `self._logger.error()`를 사용하면 UI 로그에도 표시되어 현장 운영자가 즉시 인지할 수 있다. 어느 것을 쓸지 Phase 1 구현 전에 결정하여 계획서에 명시 권장.

3. **[M1] import traceback 위치**: 현재 `traceback`은 L688에 지역 import되어 있다. Phase 1에서 파일 상단에 `import traceback`을 추가하면 지역 import를 제거하거나 그대로 둬도 무방하다(중복이지만 오류는 없음). 계획서의 "중복 import 방지" 주석은 적절하나, 기존 지역 import 제거 여부도 명시할 것.

## 종합 판정

- **구현 가능 여부**: 수정 후 가능 — Phase 1의 `_last_detection_time` 이동 위치를 수정해야 함
- **Critical 해소 필요**: C2 — `_last_detection_time`을 `try:` 블록 **내부**가 아닌 `try:` 직전(DIAG 블록 이후)으로 이동하도록 계획서 수정 필요

  - DIAG 블록 try-except 보호(Phase 1 핵심)는 설계 방향과 코드 정합성 모두 정확하며 즉시 구현 가능
  - Phase 2는 traceback 확보 없이는 진행하지 말 것 (계획서 프로토콜 준수)
  - Phase 3 (테스트 버튼 먹통)은 현재 코드(`_on_telegram_test`, `_poll_telegram_test`, `play_test_sound`)를 보면 메인 스레드 이벤트 루프가 살아있는 한 버튼 클릭은 처리됨 — freeze 상태에서 "테스트 중..." 표시 여부 확인이 핵심 단서이며 계획서 프로토콜(Q7 확인 후 진행)이 정확함
