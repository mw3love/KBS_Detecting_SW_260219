## 수정 상세 검증

### 수정 1 + 2 연계 동작 확인

수정 전 구조:
```
L283: self._last_detection_time = time.time()   ← 갱신 먼저
L284: if _detection_count % 150 == 0:
          # DIAG 블록 (예외 발생 시 함수 종료)
L403: try:   ← DIAG 예외 시 미도달
```

수정 후 구조:
```
L284: if _detection_count % 150 == 0:
          try:
              # DIAG 블록
          except:
              # 로그 후 계속
L417: self._last_detection_time = time.time()   ← DIAG 예외와 무관하게 항상 실행
L419: try:   ← 항상 도달
          # 감지 로직 전체
```

`_latest_frame is None` 조기 반환(L277)은 DIAG 블록 도달 전에 반환하므로 L417도 실행되지 않는다.
이는 의도된 동작이다 — 프레임이 없으면 감지가 불가능하므로 `detect_stale=True`가 맞다.

### 수정 3 예외 억제 로직 검증

```python
except Exception as _diag_e:
    _diag_e_type = type(_diag_e).__name__
    if _diag_e_type != getattr(self, "_diag_last_error_type", None):
        self._diag_last_error_type = _diag_e_type
        _log.error("DIAG 로깅 오류 (감지 계속): %s\n%s",
                   _diag_e, traceback.format_exc())
    else:
        _log.error("DIAG 로깅 오류 반복 (감지 계속): %s", _diag_e)
```

- 최초 예외(또는 타입이 바뀐 예외): traceback 전체 로그 → 원인 파악 가능
- 동일 타입 반복: 1줄 요약만 → 로그 파일 증가 방지
- 단, `_log.error()`는 매 30초 반복 실행됨(억제 시에도). UI 로그 위젯에 `error` 레벨 로그가 30초마다 1줄씩 계속 출력된다. 운영자가 인지할 수 있어 **긍정적 부작용**이다.

### threads=py/os 추가 로그 효과

`SYSTEM-HB` 로그에 `threading.active_count()`(Python 레벨)와 `psutil.Process().num_threads()`(OS 레벨)가 추가됨.
OS 스레드가 Python 스레드보다 많으면 네이티브 확장 스레드(sounddevice PortAudio, OpenCV 등)를 나타낸다.
장기 운영 시 OS 스레드 수가 증가 추세이면 H2(스레드 누수)의 현장 확인 수단이 된다.
